This user guide aims to explain how the system works. The explanations will follow a fictional case study to describe the steps one by one.

The case

The requirements prioritization (RP) case study aims to prioritize the requirements of a hypothetical commercial company's marketing portal that provides registered company's customers with information about the company's products. This project executed in a distributed development context where the team of stakeholders involved in RP consisted of three individuals, namely the project owner, a senior software engineer and a customer representative.

In the next development sprint of the marketing portal case study, four functional requirements are elicited to be prioritized. Their descriptions, in a high-level form, are: (i) "a registered customer uses the portal to issue an order for a company's product", (ii) "a non-registered customer uses to portal to download product brochures and catalogues", (iii) "a registered customer uses the portal to create and send to the company a direct mail asking information about a specific product" and (iv) "a registered customer uses the portal to write comments for a company's product".

The following three prioritization criteria are identified by the project manager in the marketing portal case study, namely:

(i) "importance, i.e. the value of a requirement with regard to the portal customers and the marketing position of the company". The four requirements will be ranked according to their importance by the customer representative.

(ii) "duration, i.e. the time needed to implement the requirement". The four requirements will be ranked according to their importance by The four requirements will be ranked according to their duration by the senior software engineer.

(iii) "damage, i.e., the extend of disadvantage that would result from neglecting the requirement". The four requirements will be ranked according to their damage by the project owner.

As project manager

Creating a project

First the project needs to be created. An overview of all projects created by the current project manager can be found under Projects. Here you can choose the project you want to work on currently.

The form under Create new Project gives you the opportunity to create a new project. You have to specify the title and a short description of the project. Furthermore, you have to choose criteria according to which the users will rank the requirements later. An explanation of the criteria can be found in the table below.

Please note that the criteria used in a project cannot be changed once the project is created.

Criterion Description
Importance "Importance" can consider different aspects of requirements. Examples would be the urgency of implementing it, strategic importance or the importance to the customer.
Cost "Cost" refers to the financial resources needed to implement a requirement. It usually depends on the complexity of the implementation and efforts needed for quality assurance. Other factors can also be the degree of reuse and the extent of documentation needed.
Damage "Damage" describes the extent of damage or disadvantage resulting of neglecting the implementation of the requirement. Examples can be contract penalties, reduced sales potential or instability of the system.
Duration The "Duration" requirement refers to the time needed to implement the requirement.
Risk "Risk" includes the risk that is involved when realising the requirement. Examples of risked damages would be exceeding the time schedule, resulting instabilities or low performance to the system, not satisfying the customer or even the failure of the project.
Volatility "Volatility" refers to the probability of changes to the requirement in the course of the project. Requirements with high volatility usually increase the development cost because changes have to be considered when designing the architecture of the system. Therefore it must be designed with a higher degree of flexibility which typically increases the complexity.

Creating requirements

After creating the project and setting it as the active project, requirements need to be created. These requirements will later be rated by the users assigned to the project. An overview of all already created requirements can be found under Requirements. Creating new requirements is simple.

Under Create new Requirement you have to give the requirement a title and a short description. After creating it, the requirement will appear in the overview.

Assigning users

In order to get rankings for the requirements created you need to assign users to your project. This can be done under Assign Users to active Project.

You can choose one or more users from the pool of registered users. An overview of all users assigned to the currently active project can be found under Users. There you can also unassign users.

Getting the final prioritization

The final prioritization for the requirements can only be displayed if all assigned users have completed their rankings. An overview of the given rankings can be found under Ranking. The view contains information on wether or not a user has completed the ranking.

Additionally, if there are rankings, every ranking is displayed alongside withe the criterion according to which the ranking was carried out. The tables for each ranking also shows the intutionistic fuzzy numbers that were calculated based on the given ranking of requirements.

As soon as all users completed the ranking of requirements, the final prioritization is accessible under Final Prioritization. This view shows the prioritization together with the scores of each requirement. Further information is given on the calculated weights and hesitation values for the individual rankings. For a more pleasant presentation the requirements and their scores are visualized in a barchart.

As normal user

All users except for the project manager are considered "normal" users. They are assigned to projects in order to rank the respective requirements according to a criterion of their choice.

Choosing a project

If a normal user is assigned to project, an overview of all assigned projects can be viewed under Assigned Projects. There you can choose which project to work on.

Ranking requirements

The key activity of normal users is ranking the requirements given for the active project. Ranking provides an overview of the requirements to rank. To start the ranking process head to Rank Requirements. If you have already specified a ranking, it will be displayed on the right side of the screen. Giving a new ranking changes the old one.

There you first have to choose the criterion according to which you rank the requirements. Then, you specify a rank for each of the requirements based on the criterion chosen. Keep in mind, that the ranking is positive. That means, that a ranking according to importance the requirements from the most to the least important requirement ranks. On the other side, a ranking based on cost ranks the requirements from the least to the most costly requirement. The ranks have consecutive, so it is not possible to have a ranking like 1-2-4-5. It is possible, to give the same rank to different requirements, if you see them as equal. It is also possible, to rank requirements as unknown, in case you do not know, how the requirement should be ranked according to the criterion and in regard to the other requirements.
